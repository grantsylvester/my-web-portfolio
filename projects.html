This Website

The first personal project I would like to cover is this website that you yourself are currently browsing! I decided to make this website for several reasons. First, I wanted to learn HTML, CSS and JavaScript in case I ever end up needing to design or at least understand a website or web application in my future.

Since I was interested in learning how to make a website, I decided that it would also be a cool way to make a project portfolio where I can yap about all of my interests. And here we are!

How did I make this website?

I made this website with GitHub Pages (its free). To learn how to host a website from GitHub, I followed GitHub's tutorial and made a test website. This is not that website, as the tutorial uses markdown formatting instead of HTML. To learn HTML, I followed W3 School's guide. They have a well structured online reference as well as short quizzes at the end of every chapter, which I found super helpful (and its also free). GitHub Pages requires that you push changes to main and wait for it to deploy the website again in order to see the results of your changes. In order to develop the website and see changes immediately, I installed the Live Server extension on VSCode, which let me open any HTML file in a local server by simply right clicking it.

Learning JavaScript

Another goal I had for this project was to use it as an opportunity to learn some JavaScript. I have so far accomplished this by adding the scroll-to-top button in the bottom left of your screen as well as the dropdowns buttons at the top of each project entry. Feel free to go to right click > inspect element > sources > scripts.js to see how I accomplished the creation of all of the dropdowns with one function. Basically I have an array that stores the name of every currently hidden dropdown, so it starts empty and is filled with project entry names as you hide more entries. Then my function checks if that description is in that array when you click the button, and if it is, it will show it and remove it from the array, otherwise it will hide it and add it to the array. I'm not sure if its the best way to do it, but it was my own creative solution and I am proud of it.

One more thing that I have added to the site is image galleries. As of writing this, I have added one to the Switch Dock Modification, Minecraft Redstone Designs and Project Emrick sections of this page. These have arrow buttons below that the user can use to switch between the next or previous image in the gallery, and circle buttons that the user can click to navigate directly to the image they want to see. They work by referencing and modifying the <section>Images object at the top of scripts.js to change the image source and image hover title to the desired one. To make the functions work for all future galleries, I have created an imageMaster object that contains the gallery objects. This way, all my JS functions need as an argument is the image ID (except for the selectImage function which also requires the specific destination requested by the user's choice of button).

Of course, I am not done developing this website, and I intend to learn how to make some smooth transitions next.

Switch Dock Modification

This is a project that I have had brewing in my head for a while (2 years lol). I had the idea while working at my first internship, right when I was getting into embedded systems, but when school and other jobs started to take up most of my time, I unfortunately put it on hold. Now I am ready and excited to start it up again!

The Idea

Basically, I want to modify a Nintendo Switch Dock to do a pretty Breath of the Wild (BOTW) themed light show when you dock the switch in it. I got inspiration from these cool wireless chargers I saw online. So imagine that, but on the front of a Nintendo Switch console dock. This sort of thing is very fitting, as a primary game mechanic of BOTW is the tablet that Link carries with him and uses to interract with objects and gain new skills. When interracting with these objects they usually do this sort of flashy pattern, so my hope is that this feels a little bit like bringing part of the game into the real world.

Progress

So far, I have bought an extra dock on Ebay, realized it was in better shape than mine, and dismantled mine instead. The dock has one main PCB, which houses all of the USB and HDMI ports that can be used to connect controllers, monitors, etc. How I plan to trigger the light show is to use the status light signal that is already present in the dock. The dock has a green light that turns on when the console is on and docked, so this is perfect for sensing when I place the switch in the dock.

I have also had to dust off my Fusion 360 skills since my internship with Wavelogix (see work projects). I will need to design and print some parts for the modification to look good, including the entire front face of the dock. I have so far modeled the PCB itself, but still need to model the back half of dock and design a new front half, which will house the extra electronics I will be adding.

"What extra electronics will you be adding?" you ask?. I currently plan on using a Seeed Studio Xiao microcontroller. It is ideal for this application due to its incredibly small size and low number of pins, as I won't require that many. Other than that, I still need to select a good LED to use for the display, but that shouldn't be too hard.

Minecraft Redstone Designs

In case you were not aware, the game Minecraft has a fully fleshed-out electrical system called redstone. In the game, it is usually used for small practical things such as a button to open a door or a lever to turn on the lights in your base. If you are without limitations in creative mode, however, it becomes much easier to create much larger, real world circuitry.

Growing up, this always seemed too hard to figure out, so I just played the game like a normal person instead of learning digital system design as a 10 year old (biggest mistake of my life if you ask me). Now, however, I have a bit more free time and have learned how things like Systemverilog and RTL design work, so I thought I would give it a shot.

Logic Gates

The first thing that I did was learn some basic logic gates. These are not too hard, and usually only take up 2-blocks of space. The hardest part of redstone is the fact that if you place dust (wire) next to more dust, it WILL connect, so you really have to take advantage of the third dimension when designing circuits.

D Flip-Flop

The next thing I created was a digital flip-flop, and I wanted to do it with nothing but a gate level schematic. There are lots of nifty and compact ways to make a lot of components, but I wanted to do it the primitive way first. I constructed it with only nor gates and much to my surprise, was successful on the first try. The output would only match the input if the enable signal was high and would otherwise remain at its original value.

8-Bit Register

After storing one bit, I thought,"why not store 8?". Now, however, I ditched my primitive, gate-level flip-flop and found an online example of a compact and stackable one. These work by using hoppers with outputs connected in a feedback loop. A redstone comparator checks if the hopper on the outside contains an item (1) or if it is empty (0). With clever block placement, you can make it so that the hoppers only transfer items if the enable signal is high, just like a flip-flop. This design is stackable, so all it takes to make an 8 bit register is to put 8 of these bad boys next to each other.

7-Segment Display

This is my current Minecraft project, and I have made good progress on it so far. I have created the 7 segment digit itself using redstone lamps, and I have connected 16 inputs to display 0-9 and A-F on it with a flick of the lever. The next step, which I am currently working on, is to make a 7 segment display decoder that will display the value in a 4 bit register on the display digit.

USB Transceiver

The ASIC design lab class I took was one of the most challenging but also rewarding classes I have ever taken. Did it suck all of the free time out of my life? Yes. But the feeling of making a really good Systemverilog testbench to test a module that you created with a very thorough and detailed RTL diagram and see it all work really makes it feel worth it. That or it's Stockholm syndrome, take your pick.

The final project for this course was a collaborative project where I teamed up with two other students and designed an AHB-lite enabled USB Transceiver. My specific portion of the project was the AHB subordinate device, responsible for communicating with an SoC to operate the transceiver. Unfortunately, I cannot show my RTL, testbench, module, state diagram, or waveforms for academic integrity reasons but I assure you they were beautiful.

Since I cannot show anything, I will instead recap one design challenge that I encountered and how I solved it. In the end, my subordinate device was able to transfer data to and from the USB transceiver, and the USB data was stored in a FIFO data buffer that could only communicate 1 byte to/from my device per cycle. I had to design a finite state machine to handle reads/writes of greater than 1 byte from the buffer. My initial solution was to try to stall the address phase of the transaction (bring hready low immediately), load the bytes into an intermediate register, then allow the data phase to happen once the data was available on hrdata. Unfortunately, this method contains an oversight in that I couldn't (reliably) pull hready low before the data phase. I had to stall the data phase, and due to the pipelined nature of AHB, that meant that I needed to register the old address, size, etc of the transaction because they are replaced by the information of the next transaction when the data phase begins. So, I did just that. I registered the inputs, and if the address of a transaction indicated that it was a buffer operation, my device would use those registered inputs rather than the current inputs to perform the buffer operation.

What this class/project did for me was demystify the process of how computers are designed. Its always very easy to assume that its magic performed by someone smarter than you, but understanding even a smaller ASIC like this at the register level makes the whole ordeal seem more acheivable.

Smart Home System

In my microcontrollers class, I had a final project that required me to make a device using a STM32 microcontroller. My group decided to make a prototype for a smart home system, complete with an alarm, thermostat, and an LCD display to communicate to the user any important information. The alarm system could be enabled and disabled with a keypad, and was set off by a hall effect sensor. If the alarm was triggered, a buzzer utilizing the microcontroller's PWM would sound. The thermostat consisted of a temperature sensor and a motor that served as a mock AC system. The temperature sensor was read with an ADC, and if the current temperature was higher than the target temperature (specified by the user with the keypad), the motor would turn on, as you would expect an automatic AC to work. Lastly, we used an SPI-enabled LCD display to display the alarm state, current temperature, and target temperature to the user.

JOS Project

In case you are unfamiliar, the JOS project is a popular way to teach operating systems at universities. Developed at MIT (I think), it gives the student a skeleton of an OS that, through the course of the semester, they implement many of the fundamental properties of a modern operating system. For me, the project was split into 5 labs, in which my lab partner and I set up the bootloader, the paging system for virtual memory, the user environments (processes), preemptive multitasking, and file system.

JOS is coded in C, and uses the QEMU virtual machine to run the OS on a virtual x86 architecture. This means that the project required a strong understanding of data structures in C, as well as a firm comprehension of assembly programming.

The project was as challenging as it sounds, but I left my Operating Systems class with an understanding of how computers manage many proccesses at once, how they handle memory, and how file systems work. I expect this knowledge to be very useful if I have a future in embedded systems.

Radiation Effects Testing PCBs

In the summer of 2024 I interned with Reliable Microsystems. At Rel-Micro, I was a part of the PCB design team, whose job was to create the large circuit boards that the company uses to test its radiation hardened ASICs in lab scenarios. While there, I participated in the design of 2 such PCBs, from start to finish. I made diagrams to map out the signals on the die pad of the chip, and documented all of my changes for future reference. Along with my team, we participated in multiple quality review meetings with the higher-ups of the company to ensure the PCB had no issues. We also minimized hazards such as parasitic capacitance/inductance, as these can be detrimental in a radiation testing environment.

After completing this job, I feel like I am industry ready in terms of PCB design skills. I know KiCad like the back of my hand now, and I am certain that I could pick up any other PCB software very quickly thanks to my experience at Rel-Micro. Another software that I was introduced to at Rel-Micro was Cadence Virtuoso IC design software. While none of the large projects I worked on there involved Cadence, I was exposed to the software and got to design and run simulations on a small transistor level schematic that I made.
